# Evidence Philosophy

Strategic Intelligence Agent should be read as an evidence-organizing system, not an evidence-verifying system.

## Separation Of Claims

The repository should keep these categories distinct:

| Category | Meaning |
| --- | --- |
| Facts | Statements present in the input text or local knowledge files. |
| Historical analogues | Curated cases that may resemble the current issue. |
| Historical outcomes | Simplified records of observed consequences and responses. |
| Heuristics | Deterministic rules used for routing, classification, matching, and formatting. |
| Assumptions | Conditions that make the current recommendation reasonable. |
| Interpretation | Analytical framing generated by the workflow. |
| Recommendation | Decision-support output derived from current criteria and evidence. |
| Limitations | Reasons the output may be incomplete, stale, or wrong. |

## Evidence Traceability

Each analysis run creates artifacts that support review:

```text
outputs/runs/<run_id>/
  input.txt
  analysis.json
  brief.md
  brief.txt
  agent_trace.json
  metadata.json
```

The most important artifact for audit is `analysis.json`, because it preserves the structured fields used by the dashboard and downloadable outputs.

## What Evidence Does Not Mean

Evidence in this repository does not mean:

- The system has verified a live fact.
- The historical case is identical to the current issue.
- A similar historical outcome will repeat.
- A recommendation is legally, financially, or operationally correct.
- The benchmark score is a real-world accuracy measure.

## Good Evidence Behavior

The product earns trust when it:

- Shows source origin.
- States limitations.
- Separates actions from monitoring.
- Makes assumptions visible.
- Treats historical cases as support, not proof.
- Avoids future-outcome claims.
